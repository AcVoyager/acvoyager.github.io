记配置VPS中遇到的知识点

Posted on 2021-05-23 Edited on 2021-07-05 In 中文, English Views: Disqus:

回国的第一篇blog.

持续更新…

本篇文章不是讲如何配置VPS的，网上有很多类似教程。本篇记述在配置过程中，对于一些技术/知识点/细节的学习和总结。

SSH (Secure Shell)

参考链接：Understanding the SSH Encryption and Connection Process

发动SSH请求的设备叫client，接收SSH的叫server。整个SSH session分为两个阶段，首先是协商加密方式，然后是验证用户（client）。在整个过程中，对称加密，非对称加密和Hashing都有用到。

阶段一: Negotiating Encryption for the Session

此阶段的目的其实很容易理解：生成一个后续所有交流都需要用的symmetric key。这个目的和HTTPS一开始的加密目的是一样的，不过过程有所不同。

核心是Diffie-Hellman算法。在参考链接中，第六步至关重要：

The receiving entity uses their own private key, the other party’s public key, and the original shared prime number to compute a shared secret key. Although this is independently computed by each party, using opposite private and public keys, it will result in the same shared secret key.

请注意，阶段一最后回生成一个shared secret，这个shared secret最厉害的地方就在于它们是在各自的本地计算出来的，但是值都一样，因而可以被用来为后续所有的交流进行加密。这个secret key，与阶段二用户验证无关。该secret是一个对称加密密钥。

阶段二: Authenticating the User’s Access to the Server

实际上阶段二才是人们常接触到的阶段：是输入密码还是提供credential？
密码是最简单的一种方式，但是安全性不够高，可能会被暴力破解。

另一种较为推荐的方式也就是人们熟知的SSH key pairs，公密钥对。其安全性在于，公钥加密的数据只能被私钥解析，而私钥无法通过公钥推断出来。

此过程中，server保有公钥，而client使用私钥解析公钥发送来的信息（通常是一个数字），这也是为什么在使用此方式的时候，ssh需要指明一个private secret的原因:

解析之后，server用MD5计算出发来的数字的hash value，然后用阶段一的shared secret发还给server。server用shared secret解析出MD5值，并自行用MD5计算发送的数字的hash value。如果对上了，说明client是private key的持有者，那么该用户被验证成功。

因此，一般我们买了台VPS后，要不通过VPS服务商提供的系统上传一个key pair的公钥，要不就是先用password登陆，然后自行添加公钥。Vultr就属于前者，当然后续添加也没问题。
